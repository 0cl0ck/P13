Voici les informations promises pour les trois grandes parties :

- La mise à jour du document **“Business Requirements”**.

  Je t’en fournis aussi un modèle, que tu trouveras en pièce jointe.  
  Ce document sert à évaluer la conformité du projet, et doit donc définir des listes de contrôles qui nous permettront de vérifier que l’implémentation respecte bien le dossier de définition d’architecture.

## Étapes

### Prérequis
Avoir lu et compris les exigences envoyées par la CEO.

### Résultat attendu
Avoir complété le document **“Business Requirements”**.

### Recommandations
- Examiner avec attention le document existant.  
- Identifier les éléments implicites communs à toutes les applications web, et vérifier s’ils sont tous déjà présents dans le document.  
- Ajouter ceux qui manqueraient, selon votre analyse.

### Points de vigilance
- Prendre soin d’assurer une structure cohérente tout au long du document.

### Ressources
- Suivre le cours OpenClassrooms **Réalisez un cahier de charges fonctionnel**.

---

### Prérequis
Avoir complété le document **“Business Requirements”**.

### Résultat attendu
- Plusieurs diagrammes qui représentent l’architecture technique.  
- Avoir identifié les technologies compatibles avec l’architecture technique retenue.

### Recommandations
- Utiliser des diagrammes UML (diagramme de composants, diagramme de déploiement).  
- Faire des choix basés sur les technologies que vous maîtrisez, pour être plus à l’aise dans vos justifications.

### Points de vigilance
- Ne pas tomber dans le piège de laisser des zones de flou dans la modélisation.  
- Prendre soin de respecter les standards de modélisation existants.

### Ressources
- Lire ces tutoriels d’UML Lucidchart :  
  - Qu’est-ce qu’un diagramme de composants UML ?  
  - Qu’est-ce qu’un diagramme de déploiement UML ?

### Prérequis
Avoir complété le document **“Business Requirements”**.

### Résultat attendu
- Au moins un schéma de base de données aux normes (selon le type de base de données retenu).  
- Le ou les fichiers pour mettre en place la base de données (par exemple scripts SQL ou fichiers JSON).

### Recommandations
- Utiliser une approche **DDD (Domain Driven Design)**.  
- Modéliser d’abord le domaine via un diagramme de classes UML, puis le convertir en schéma de base de données.

### Points de vigilance
- Respecter les standards de modélisation (par exemple si BDD relationnelle : les formes normales).

### Ressources
- Suivre ces cours OpenClassrooms sur la modélisation :  
  - Appliquez le principe du Domain-Driven Design à votre application  
  - Modélisez vos bases de données
